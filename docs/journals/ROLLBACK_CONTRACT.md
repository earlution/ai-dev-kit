# RW Rollback / Compensation Contract

**Applies to:** Release Workflow (RW) execution artifacts.  
**Goal:** Explicitly define what can be automatically reverted, what requires manual reconciliation, and what is forbidden.

---

## 1. Principles

1. **Conservatism first.** When in doubt, mark a surface as manual-reconcile.
2. **No destructive remote ops.** Force-push, remote tag deletion, and history rewrite are never suggested automatically.
3. **Immutability of journals.** Rollback contracts are derived from journals; journals themselves are never modified.
4. **Operator is the final authority.** Automatic revert steps are advisory; the operator must confirm before executing.

## 2. Automatic Revertible Surfaces

These can typically be undone with no remote impact and no data loss risk.

| Surface | Example | Revert Command | Safety Notes |
|---------|---------|---------------|--------------|
| Unstaged working-tree file | `src/ai_dev_kit/version.py` modified but not `git add`ed | `git checkout -- <file>` | Reverts to HEAD. Ensure you don't need the changes. |
| Staged but uncommitted file | `CHANGELOG.md` staged via `git add` | `git reset HEAD <file>` | Unstages only; working tree keeps changes. |
| Untracked file | Temporary `.tmp` generated by script | `rm <file>` | Safe if file is not in `.gitignore` by design. |
| Uncommitted tag | Local tag created but not pushed | `git tag -d <tag>` | Only safe if tag was never pushed. |

## 3. Manual Reconciliation Surfaces

These require operator review because they may affect shared history or remote state.

| Surface | Example | Recommended Action | Safety Notes |
|---------|---------|-------------------|--------------|
| Committed change | Version bump committed locally | Amend with `git commit --amend` or create a follow-up commit | If already pushed, prefer follow-up commit over amend. |
| Pushed commit | Commit pushed to `origin/dev` | Create a revert commit (`git revert <sha>`) or a follow-up fix | Never force-push to rewrite shared history. |
| Pushed tag | Annotated tag `v0.6.1.38+3` pushed | Do **not** delete remote tag. Create a new corrected release. | Remote tag deletion can break consumers and CI. |
| Merged release branch | `dev` merged to `main` | Revert the merge commit if needed (`git revert -m 1 <sha>`) | Coordinate with team; may affect downstream builds. |
| GitHub Release | Release created via API | Edit or draft a new release; do not delete existing release. | Deletion removes release notes and asset links. |

## 4. Forbidden Actions

The following actions must **never** be suggested or executed automatically by the recovery system:

- `git push --force` or `git push -f`
- `git reset --hard`
- `git clean -fd` (destructive removal of untracked directories)
- `git branch -D` on a shared branch
- `rm -rf` on any tracked project directory
- Remote tag deletion (`git push origin :refs/tags/<tag>`)
- Remote branch deletion (`git push origin :<branch>`)

If any of these commands appear in a step's command log, the rollback contract flags them as **forbidden** and requires explicit operator override.

## 5. Compensation (When Rollback Is Impossible)

Some changes cannot be rolled back. In these cases, the contract prescribes **compensation** — a forward-fixing action:

| Irreversible Change | Compensation Action |
|---------------------|---------------------|
| Pushed bad tag | Create a new release with incremented build number and deprecate the old one. |
| Published bad package | Yank or deprecate the package version; publish a corrected one. |
| Released changelog with error | Append a correction entry in the next release; do not rewrite published history. |
| Kanban board updated with wrong status | Run UKW to reconcile status manually; do not edit historical board entries. |

## 6. Safety Guardrails in Code

The `RollbackContract` class enforces these rules programmatically:

- `FORBIDDEN_COMMANDS` set is checked against every logged command.
- `_is_committed()` uses `git diff --quiet HEAD` to determine if a file is already committed.
- `manual_reconcile` surfaces always include a `safety_notes` field explaining why operator review is required.

## 7. Operator Checklist Before Rerun

Use this checklist after a failed run and before re-invoking RW:

- [ ] Recovery report has been read.
- [ ] Auto-revertible files have been reverted (if desired).
- [ ] Manual-reconcile files have been reviewed.
- [ ] No forbidden actions are present in the rollback contract (or they have been explicitly acknowledged).
- [ ] Root cause of the original failure has been addressed.
- [ ] Git status is clean or intentionally dirty with only the desired changes.
- [ ] The same task token will be used for the rerun.
