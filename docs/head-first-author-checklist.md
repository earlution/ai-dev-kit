# Head First Author Checklist

**Quick Reference Checklist for Authors**

Use this checklist when drafting chapters to ensure consistency with the Head First style guide.

---

## Book-Level Checks

- [ ] **Title** follows "Head First AI-Assisted Development: Scaling Vibe Coding into Professional Workflows" pattern
- [ ] **Audience** clearly defined as technical professionals (not absolute beginners)
- [ ] **Parts**: 4 Parts, each reflecting progression from practice to professional system
- [ ] **Part names** are progression-oriented and professional (not clever or abstract)

---

## Chapter-Level Checks

Each chapter should have:

- [ ] **One central theme or problem** that the chapter tackles
- [ ] **Engaging introduction (1–2 pages)** that:
  - Hooks the reader with a real problem or scenario
  - States what the chapter covers
  - Explains why it matters for building professional systems
- [ ] **3–6 sections** with:
  - Clear, outcome-based headings (describe what reader can do/decide)
  - At least one hands-on exercise or activity
  - Visual elements (diagrams, code examples, illustrations)
- [ ] **Active Learning Elements**:
  - At least one **exercise** (hands-on framework implementation)
  - 1–2 **puzzles** (problem-solving scenarios)
  - Multiple **code examples** (real framework code from dev-kit)
  - **Case studies** where appropriate (real project examples)
- [ ] **Callouts** distributed throughout:
  - 2–3 TIPs
  - 1–2 WARNINGs
  - 1–2 REMEMBERs
  - 1–3 TECHNICAL STUFF blocks (technical depth expected)
- [ ] **"What You've Learned" wrap-up** (1–2 paragraphs) summarizing key takeaways and connecting to next chapter

---

## Section-Level Checks

Each section should:

- [ ] Be **focused and engaging** (avoid sections longer than 3–4 pages without visual breaks)
- [ ] Have a **descriptive heading** that describes outcomes or decisions
- [ ] Use **visual elements** (diagrams, code examples, illustrations) to support learning
- [ ] Use **lists** to break up long explanations:
  - Numbered lists for procedures/sequences
  - Bulleted lists for options/examples/pros-cons
  - Code blocks for real framework code
- [ ] Include **concrete examples** that are realistic and professional

---

## Style & Tone Checks

- [ ] Written in **second person** ("you", "your")
- [ ] Assumes **technical professional background** (Git, programming, command line familiarity)
- [ ] Uses **technical language appropriately**—readers expect and understand technical terms
- [ ] **Clear paragraphs** (typically 3–6 sentences, can be longer for technical explanations)
- [ ] **Clear sentences**—technical precision without oversimplification
- [ ] Uses **contractions sparingly**—maintain professional tone while staying conversational
- [ ] Prefers **active voice**: "You implement the framework" not "The framework is implemented"
- [ ] **Technical jargon** is:
  - Used when appropriate (readers expect it)
  - Defined or explained when introducing new concepts
  - Used consistently throughout
- [ ] **Engaging tone**—conversational, respectful of technical competence
- [ ] **Problem-solving focus**—shows how to solve real professional challenges

---

## Active Learning Checks

### Exercises

- [ ] At least **one exercise per chapter**
- [ ] Exercises are **hands-on** (readers implement something)
- [ ] Exercises use **real framework code** from dev-kit
- [ ] Exercises are **step-by-step** with clear instructions
- [ ] Exercises connect to **real professional scenarios**

### Puzzles

- [ ] 1–2 **puzzles per chapter**
- [ ] Puzzles present **real-world problems**
- [ ] Puzzles have **multiple solution paths** or require problem-solving
- [ ] Puzzles reinforce **key concepts** from the chapter

### Code Examples

- [ ] Multiple **code examples per chapter**
- [ ] Code examples are **real framework code** from dev-kit
- [ ] Code examples are **complete enough** to understand context
- [ ] Code examples are **commented** for clarity
- [ ] Code examples are **tied to exercises** or activities

### Visual Learning

- [ ] At least **one visual element per chapter** (diagram, illustration, etc.)
- [ ] Visual elements support **complex concept understanding**
- [ ] Visual elements include:
  - Architecture diagrams
  - Workflow diagrams
  - Sequence diagrams
  - System diagrams
  - Code structure diagrams

### Case Studies

- [ ] 1–2 **case studies per part** (where appropriate)
- [ ] Case studies show **real project examples** (200K+ LOC projects)
- [ ] Case studies include **actual challenges and solutions**
- [ ] Case studies demonstrate **scale and complexity** appropriate to professional context

---

## Content Handling Checks

### Procedures (How-tos)

- [ ] Use **numbered lists** for multi-step tasks
- [ ] Each step starts with a **verb** (imperative form)
- [ ] Each step describes **one clear action**
- [ ] Action comes first, then brief explanation if needed
- [ ] Include **code examples** or **diagrams** where helpful

### Concept Explanations

- [ ] Follows the rhythm:
  1. What it is (clear technical definition)
  2. Why it matters / where it shows up (professional relevance)
  3. One concrete example (realistic scenario)
  4. Hands-on exercise (readers implement it)
- [ ] Includes **architectural considerations**
- [ ] Shows **trade-offs and design decisions**
- [ ] Connects to **real-world patterns**

### Examples

- [ ] Examples are **realistic** and grounded in professional scenarios
- [ ] Examples use **real framework code** from dev-kit
- [ ] Examples show **actual project structures** and patterns
- [ ] Examples include **complexity appropriate** to professional context
- [ ] Examples tie directly to **actions readers can perform** in their own projects

---

## Callout Checks

### TIP
- [ ] Marks shortcuts, optimizations, or professional best practices
- [ ] Format: `TIP: [text]` at start of paragraph/line

### REMEMBER
- [ ] Marks key concepts or architectural principles reader must not forget
- [ ] Format: `REMEMBER: [text]` at start of paragraph/line

### WARNING
- [ ] Marks risks, common mistakes, or things that can break systems or waste time
- [ ] Format: `WARNING: [text]` at start of paragraph/line

### TECHNICAL STUFF
- [ ] Marks optional deeper explanations or architectural details (can be skipped)
- [ ] Format: `TECHNICAL STUFF: [text]` at start of paragraph/line

---

## What to Avoid

- [ ] Talking down to readers (they're technical professionals)
- [ ] Oversimplifying complex topics
- [ ] Abstract theory without practical application
- [ ] Dense, unbroken paragraphs without visual breaks
- [ ] Impersonal language ("the developer", "one")
- [ ] Patronizing tone
- [ ] Examples that are too simple for professional context
- [ ] Missing active learning elements (exercises, puzzles, code examples)

---

## Final Review Questions

Before submitting a chapter, ask:

1. **Does this chapter** include hands-on exercises that readers can implement?
2. **Are visual elements** (diagrams, illustrations) included to support learning?
3. **Is the code** real framework code from dev-kit?
4. **Does the tone** respect readers' technical competence?
5. **Would a technical professional** feel "I can build this" after reading?
6. **Are examples** realistic and professional?
7. **Are new concepts** defined clearly while using appropriate technical language?
8. **Are callouts** placed near relevant content, not randomly?
9. **Does the chapter** include "What You've Learned" wrap-up?
10. **Are exercises** tied to real professional scenarios?

---

## Quick Style Reference

- **Voice**: Second person ("you")
- **Tone**: Conversational, engaging, respectful of technical competence
- **Sentences**: Clear, direct, technically precise
- **Paragraphs**: 3–6 sentences (can be longer for technical explanations)
- **Lists**: Numbered for procedures, bullets for options
- **Code**: Real framework code from dev-kit, in code blocks with syntax highlighting
- **Callouts**: TIP, REMEMBER, WARNING, TECHNICAL STUFF
- **Headings**: Outcome-based, professional, not clever
- **Active Learning**: Exercises, puzzles, code examples, visual elements required
